# Alchemy-RSS: 技术深度解析与演进策略 (Technical Deep Dive)

## 1. 抓取逻辑分析 (Current Fetching Logic)

### 1.1 数据提炼深度
目前系统采用**“轻量索引”**策略。
*   **字段提取**：仅提取 `title`, `link`, `date` 三个核心字段。
*   **摘要与正文**：为了保持 Git 仓库的性能，系统主动抛弃了 RSS 源中的 `content` 和 `contentSnippet`。
*   **多媒体处理**：目前处于“无图模式”，不处理 `enclosure` 或正文中的图片资源。

### 1.2 技术栈评估
*   **解析器**：使用 `rss-parser`。
    *   **优点**：轻量、稳定、标准适配性好。
    *   **局限**：仅支持静态 XML 解析，无法处理动态 JavaScript 渲染的页面（如 SPA、Twitter 等）。
*   **去中心化存储**：利用 Git Commit 历史作为版本化数据库，实现了天然的增量更新机制。

## 2. 瓶颈与挑战 (Bottlenecks & Challenges)

*   **动态抓取缺失**：面对现代 Web 应用（无 RSS 接口、JS 渲染），目前的架构无法直接处理。需考虑引入无头浏览器（Puppeteer）或第三方代理（如 RSSHub）。
*   **信息密度低**：由于缺乏摘要，用户必须点击外链才能了解内容，不便于快速扫描。
*   **依赖老化**：`rss-parser` 虽可用，但版本较老，需评估更新到最新版以获取更好的类型支持和性能。

## 3. 演进方案：Google PM 视角 (Future Strategy)

如果由 Google 工程团队主导，本项目应从“信息搬运工”转向“智能知识中台”。

### 3.1 语义化提炼 (Semantic Intelligence)
*   **向量化索引**：引入 Embedding 技术，将文章标题和摘要转化为向量，支持语义搜索而非简单的关键词匹配。
*   **AI 降噪 (Deduplication 2.0)**：利用 LLM 识别不同源中的重复内容，实现自动聚类，仅推送质量最高、信息最全的版本。

### 3.2 质量分级系统 (Quality Scoring)
*   **权威度模型**：根据历史点击率和来源权重，为文章自动评分。
*   **炼金闭环**：建立用户反馈循环，自动优化展示排序。

### 3.3 增强型数据流水线
*   **摘要生成**：在抓取阶段调用轻量级模型（如 Gemini Flash）生成 100 字摘要。
*   **多模态增强**：自动提取文章首图并进行压缩存储，提升阅读体验。
*   **MCP 协议集成**：使 Alchemy 成为 AI Agent 的标准插件，让大模型能直接读取并检索本站的精选数据。

## 4. 存储架构的演进 (Storage Architecture Evolution)

如果引入摘要内容，数据量将呈几何倍数增长，Google 工程师会从以下维度重新审视存储方案。

### 4.1 Git 作为存储的瓶颈
*   **Git Bloat**：Git 擅长处理文本 Diff，但不擅长频繁更新的大型 JSON 文件。包含大量摘要的单文件会导致仓库体积迅速膨胀，拖慢 CI/CD 速度。
*   **结论**：Git 适合作为“索引库”，但作为“正文库”需要配合分片或外部存储。

### 4.2 存储优化策略 (10x Design)
*   **数据分片 (Sharding)**：将 `links.json` 按月份或季度进行分片（如 `data/archives/2026-01.json`）。仅在主索引中保留最近 30 天的热数据，历史数据进行物理隔离。
*   **元数据与内容解耦**：
    *   **Index (Git)**: 存储标题、链接、哈希 ID。
    *   **Blobs (Object Storage)**: 将摘要正文存储在 Cloudflare R2 或 S3 中，API 根据 ID 按需请求。
*   **内容寻址存储 (CAS)**：对内容进行哈希运算，实现全局去重，多源共享同一份内容块。

### 4.3 落地建议
在不引入外部数据库的前提下，建议采用**“按月分片 + 冷热分离”**的 Git 存储方案，以维持 Serverless 架构的优雅。
